## Slide 3 — Core Thesis & Running Case

**Data Science is the workflow that turns media data into defensible decisions.**

Students move through the full cycle, every week, on one running case:

**Question → Data → Cleaning → EDA → Statistics → Modelling → Evaluation → Interpretation → Recommendation**

**The running case (semester-swappable):**
A campaign or platform-performance dataset with audience segments, creative variants, impressions/clicks/conversions, engagement metrics, channel information, and a recommendation or targeting component.

**The driving question:** *Which media or marketing decision is justified by this data — and which is not?*

The professional target is media/data professionals who can run an applied workflow, evaluate uncertainty and bias, and translate findings into defensible recommendations — not ML engineers.

*Speaker notes:* "The running case anchors the module in concrete data work across all five weeks. Students don't switch datasets between modules — they live with one realistic campaign dataset and build deepening competence around it. The cycle from question to recommendation is what makes this Data Science rather than data literacy. The case can be swapped each semester without rebuilding the module."

## Slide 4 — Five-Week Roadmap

| Week | Theme | Core question | Output |
|---|---|---|---|
| **1** | Workflow for Media & Marketing | What makes a media problem a data-science problem? | Problem framing + data audit |
| **2** | EDA, Data Quality & Features | What does the data show, hide, distort? | EDA notebook/report |
| **3** | Statistical Reasoning & Experimentation | Which claims are statistically defensible? | Measurement claim brief |
| **4** | Machine Learning for Media Decisions | Can we predict, segment, recommend responsibly? | Model evaluation + live defense |
| **5** | Decision Communication & Ethics | What should a team or client do next? | Final data story + reflection |

*Speaker notes:* "The weekly structure follows the logic of a real data-science project. Week 1 frames the problem and audits the data — many DS failures start before modelling, in a vague question or a wrong target variable. Week 2 is hands-on EDA, with the understanding that platform metrics already contain assumptions. Week 3 is the statistical core — when is a difference meaningful, when is causal language dangerous in media analytics. Week 4 brings simple ML workflows: train/test split, baselines, classification or clustering, model evaluation metrics, feature importance. Week 5 closes with decision communication — what the analysis supports, what it doesn't, how AI was used, whether the workflow is reproducible. Each week has five competency-oriented learning objectives mapped to the official module goals."

## Slide 5 — Self-Study Architecture

**The self-study phase is active production, not passive consumption.**

**Each week follows a four-part loop (≈26h):**

→ **Input** — Microlearning units (15–30 min), avatar-narrated, modular, swappable each semester

→ **Guided practice** — Hands-on work in notebooks with the running case

→ **Applied task** — Notebook, brief, dashboard, model evaluation, or data story

→ **Mastery-oriented checkpoint** — Repeatable formative feedback

**Theoretical anchor:** Flipped Classroom 2.0 — asynchronous time for foundational learning, synchronous time reserved for higher-order cognitive work.

**Adaptive synchronous support:** statistics week receives doubled virtual classroom time.

*Speaker notes:* "Because 130 of 150 hours are self-study, this phase must be carefully designed. The four-part loop, repeated weekly, gives students a predictable rhythm. The microlearning format is compatible with the SPARKS avatar production format — short, scriptable, modular units that can be updated each semester without rebuilding the course. Every week produces something concrete: a data audit, an EDA notebook, a statistical brief, a model evaluation, a final data story. The synchronous time is concentrated where students need it most — statistics in Week 3 gets two surgery sessions instead of one."

---

## Slide 6 — Technical Depth: One Workflow, Adjustable Depth

**Required for all students:**
- Work with structured datasets in notebooks
- Perform EDA on the running case
- Interpret statistical claims with uncertainty
- Use or evaluate a simple ML workflow
- Understand model evaluation metrics
- Communicate uncertainty and limitations

**Extension for technically stronger students:**
- Compare alternative models
- Tune features and validate further
- Improve reproducibility
- Add explainability components
- Build a more advanced dashboard or notebook

**Same workflow. Same evidence standards. Different depth of technical implementation.**

*Speaker notes:* "Media & Data students arrive with uneven coding skills. Rather than splitting them by program — which doesn't make sense in a programme-specific module — I differentiate by technical depth within one shared workflow. Everyone must complete the full data-science reasoning cycle and meet the same evidence standards. Technically stronger students can go further in implementation. This keeps the cohort working on the same problem with the same standards while respecting different starting points."

## Slide 7 — Assessment Concept

**Core principle:** *AI is allowed where professional judgment is assessed. AI is restricted where personal competence must be verified.*

(Two-Lane model: Sydney; AI Assessment Scale: Perkins et al., 2024)

**Four continuous performance records:**

| # | Week | Format | AI use |
|---|---|---|---|
| **1** | End of W2 | Data audit + EDA notebook | Allowed, documented |
| **2** | End of W3 | Statistical reasoning brief | Allowed, documented |
| **3** | End of W4 | Live defense (on-site) | **Restricted** |
| **4** | End of W5 | Final data story / decision memo | Allowed, documented |

*Speaker notes:* "Four assessments, four formats, four cognitive demands — and an honest answer to AI tools in 2026. The Week 2 EDA gives early diagnostic information. The Week 3 statistical brief verifies the foundation before the on-site phase. The Week 4 live defense is the assured component — students must personally explain what they did, why it is valid, and where the limits are. The Week 5 final data story closes with professional communication. AI use is allowed in digital submissions but must be documented and reflected on. The frameworks I reference are well-established: Two-Lane assessment from Sydney, the AI Assessment Scale from Perkins and colleagues."

---

## Slide 8 — The Data Science Studio (On-Site, Week 4)

**Day 1 — Analyze, Model, Evaluate**
- Morning: dataset walkthrough; data quality and feature discussion; baseline model logic
- Afternoon: team-based model evaluation, metric choice, error analysis
- Closing: each team documents one concrete failure mode or bias risk

**Day 2 — Defend, Critique, Translate**
- Morning: peer review across teams; methodological critique
- Midday: discussion — when is a model good enough to inform a media decision?
- Afternoon: live defense to a non-technical audience — recommendation, uncertainty, limitations

**The narrative arc:** analyze → model → critique → defend.
Students defend the full workflow: problem framing, data prep, statistical assumptions, model choice, evaluation metrics, limitations, recommendation.

*Speaker notes:* "Sixteen hours on-site is precious. I use them for what asynchronous formats and AI avatars cannot do: collaborative model evaluation with real-time methodological critique, peer review, and judgment discussions that only work face-to-face. By Week 4, students have framed the problem, explored the data, and worked through statistical reasoning — the on-site phase is where they integrate everything into a defended workflow. The live defense at the end is the assured assessment: students must explain their own work, including its limits, under questioning from a non-technical audience."

## Anticipated Questions

**Q: Is this still a MIND module if it is so applied?**
> "Yes. The MIND contribution is methodological — evidence, validity, statistics, modelling, uncertainty, responsible interpretation. The application context is media and marketing; the intellectual core is scientific reasoning with data."

**Q: Why not make it more technical?**
> "Because the professional target is Media & Data, not ML engineering. The module is technically concrete — students do the full workflow including ML — but the core competence is applying data science to media decisions, not building production systems."

**Q: How do you handle uneven coding skills?**
> "One shared workflow with adjustable technical depth. Everyone completes the full reasoning cycle and meets the same evidence standards. Stronger students extend the technical implementation; all students must justify their evidence and defend their recommendation."

**Q: How do you verify understanding when AI tools are allowed?**
> "Through the live on-site defense in Week 4. AI is permitted in digital submissions if documented, but students must personally explain their data, methods, model choices, assumptions, limitations, and recommendation — under questioning."

**Q: Why is the on-site phase in Week 4, not Week 3 for statistics?**
> "Three reasons. First, statistics is best learned in repeated short sessions — two virtual surgeries in Week 3 serve that better than a two-day intensive. Second, the on-site Studio in Week 4 integrates statistics, modelling, and communication — statistics is verified in the workflow, not in isolation. Third, putting the on-site phase at the climax creates a clearer arc: frame, explore, reason, then integrate and defend."
